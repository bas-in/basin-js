# basin-js — JavaScript / TypeScript SDK

Supabase-shaped client for the Basin platform. Like `@supabase/supabase-js`:
one entry point (`createClient`), namespaces under it for auth, the
table query builder, storage, edge functions (when those land), and
realtime (when basin v0.2 ships channels).

```js
import { createClient } from '@bas-in/basin-js'

const basin = createClient(BASIN_URL, BASIN_ANON_KEY)

await basin.auth.signInWithPassword({ email, password })
const { data, error } = await basin
  .from('products')
  .select('id, name, price')
  .eq('active', true)
  .limit(10)
```

## Tier 2 — Postgrest-shaped query builder (`from()`)

The biggest piece. Matches supabase-js / postgrest-js operator surface.
The SDK calls basin-engine DIRECTLY at `/rest/v1/:table` — basin-cloud
is not in the request path.

- [x] `src/postgrest/builder.ts` — chainable builder returning a thenable — **landed (engine-direct)**. Class implements `PromiseLike<PostgrestResponse<T>>`; `.single()` / `.maybeSingle()` cast to `PostgrestSinglePromise<T>` for the narrowed `T | null` shape, `.csv()` / `.geojson()` / `.explain()` to `PostgrestStringPromise` for raw-text return. `#execute()` runs in `then()`, builds the URL with the query-string accumulator + the Prefer/Accept header chain, handles 204/text/JSON/envelope/flat shapes uniformly. URL target swapped from the cloud's `/v1/rest/v1/:table` to the engine's `/rest/v1/:table` (drop the `/v1` prefix). Module-level `unwrapPostgrestBody` keeps the tolerant envelope unwrap so both `{data, error}` and flat PostgREST responses work.
- [x] `.select(columns, { count? })` — column list, count strategy — **landed**. `select` sets the `select=` param + records the count strategy. `Prefer: count=exact|planned|estimated` header attached when `count` is set; total parsed from `Content-Range: 0-9/N` header on response.
- [x] Filter ops: `.eq`, `.neq`, `.gt`, `.gte`, `.lt`, `.lte`, `.like`, `.ilike`, `.is`, `.in`, `.contains`, `.containedBy`, `.rangeGt`, `.rangeGte`, `.rangeLt`, `.rangeLte`, `.rangeAdjacent`, `.overlaps`, `.textSearch`, `.match`, `.not`, `.or`, `.filter` — **all 23 implemented**. Each filter pushes `column=op.value` onto the URLSearchParams accumulator; `URLSearchParams.toString()` handles URL-encoding on render so PostgREST's reserved delimiters (`.`, `,`, `(`, `)`) survive the trip. Special encodings: `cs.{a,b}` / `cd.{a,b}` / `ov.{a,b}` for array containment; `fts(config).query` for full-text search with optional config; `not.<op>.value` prepend for negation; `or=(...)` as a top-level filter group with optional `foreignTable.or=(...)` scoping; `.match({k:v,...})` short-circuits to N `eq` calls.
- [x] Modifiers: `.order`, `.limit`, `.range`, `.single`, `.maybeSingle`, `.csv`, `.geojson`, `.explain` — **all 8 implemented**. `.single()` + `.maybeSingle()` set `Accept: application/vnd.pgrst.object+json` and narrow the return type to `T | null`; `.csv()` sets `Accept: text/csv`, `.geojson()` sets `Accept: application/geo+json`, `.explain()` sets `Accept: application/vnd.pgrst.plan[+json]` — all three return the raw text body as `data: string`. `.single()` errors on empty array (not_found) or multi-row (invalid_response); `.maybeSingle()` tolerates empty (data: null). `.range(from, to)` translates to `offset=from&limit=to-from+1` matching supabase-js semantics.
- [x] `.insert(rows, { upsert? })` — single + bulk — **implemented**. POSTs the rows as JSON body; `upsert: true` sets `on_conflict=*`. Accepts both `Partial<T>` and `Partial<T>[]` so callers can pass one row or N without a wrapping array.
- [x] `.update(values)` + `.upsert(rows, { onConflict })` — **implemented**. `update` PATCHes the values with the filter chain in the query string (filters become the WHERE clause). `upsert` POSTs with `on_conflict=<column>` so PostgREST chooses ON CONFLICT DO UPDATE semantics.
- [x] `.delete()` — **implemented**. DELETE with the filter chain in the query string. 204 No Content (the common `return=minimal` case) collapses to `{data: null, error: null}`; 200 with rows decodes via the same envelope path as select.
- [x] `.returning('representation' | 'minimal')` — **implemented**. Sets `Prefer: return=representation` (default) or `Prefer: return=minimal` (skips response body). When combined with `.select(..., {count})` the Prefer header carries both directives joined by comma (`return=representation,count=exact`).
- [x] RLS context — auth header from current session is auto-attached — **implemented**. `#buildHeaders()` calls `deps.auth.getSession()?.access_token` and injects `Authorization: Bearer <at>` when a session exists. The anon `apikey` header stays unconditionally — basin-rest reads both (apikey identifies the project; bearer identifies the user for RLS). Unit test confirms post-`signInWithPassword` queries carry the new bearer; queries before sign-in carry only `apikey`.
- [x] Vector ops (basin extension): `.vectorSearch(column, query, { distance, limit })` — **stubbed as `not_implemented`**. The method records the call but `#execute()` short-circuits with a `BasinError("not_implemented", ...)` before the network. Lands when basin-rest grows the `<->`/`<=>` operator surface; app code can be written against the call site today and flip on with no SDK upgrade. JSDoc on the method documents the deferred state.
- [x] Snapshot semantics for the engine's MVCC: `.asOf(snapshotId)` — **stubbed as `not_implemented`**. Records the snapshot id; `#execute()` surfaces `BasinError("not_implemented", "AS OF SNAPSHOT lands in basin v0.2")`. Same flip-on-later pattern as `.vectorSearch`.
